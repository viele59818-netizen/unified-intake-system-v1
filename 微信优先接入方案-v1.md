# 微信优先接入方案 v1

## 核心判断

当前客户沟通主阵地在微信，因此输入端应以微信为优先来源。

飞书更适合：

- 做客户版文档
- 做协作和共享
- 做结构化沉淀

所以更合理的链路不是：

- 微信直接变成飞书

而是：

- 微信负责产生原始输入
- 中转层负责收集
- 飞书负责汇总和协作
- Obsidian 负责内部沉淀

## 先说结论

### 1. 微信个人聊天内容，不能指望官方“自动直连”

就当前公开可行的官方能力来看，个人微信并不是一个适合直接自动化读取私聊内容的正式开发入口。

更准确地说：

- 个人微信不是面向这类自动化工作流开放的标准接口平台
- 小程序也不能直接读取你任意个人聊天里的历史消息
- 所以“微信里来一条消息，系统自动读到，再自动转飞书群”这条路，不适合作为第一版主方案

### 2. 第一版最可行的是“半自动中转”

最现实、最稳的方式是：

- 微信里发现重要内容
- 通过一个最小中转动作送入系统
- 后面再由系统和 AI 接着处理

### 3. 飞书仍然保留，但位置后移

飞书不做微信的原生入口，而做：

- 汇总池
- 对外文档
- 协作层

## 推荐的三层入口

### 入口 A：微信原始来源

这是高频来源。

典型内容：

- 客户聊天
- 语音转文字
- 截图
- 视频
- 图片
- 临时需求说明

### 入口 B：中转池

这是第一版真正应该设计好的层。

推荐 2 种中转方式：

1. 本地固定目录
2. 飞书收集群

### 入口 C：正式沉淀

- Obsidian：内部知识
- 飞书文档：客户版和协作版
- 滴答清单：执行动作

## 第一版最推荐方案

### 方案 1：微信 -> 本地目录 -> 我来处理

这是当前最稳、最容易落地的方案。

做法：

- 微信里看到重要内容
- 截图、导出、保存视频、复制文字
- 丢到一个固定目录
- 你来一句“处理这个专题”
- 我直接读取目录内容并整理

优点：

- 不依赖微信开放接口
- 不依赖飞书权限立即打通
- 图片、视频、PDF、网页截图都能放
- 最稳定

缺点：

- 仍需要一个保存动作

### 方案 2：微信 -> 飞书收集群 -> 我来处理

做法：

- 微信里复制文字或截图
- 手动发到飞书收集群
- 等权限打通后，我从飞书读取

优点：

- 更接近最终协作形态
- 手机上转发很顺

缺点：

- 微信不能一键直达飞书
- 还是要中转
- 当前飞书应用权限还没放行

### 方案 3：微信 -> 自建中转服务 -> 飞书群

做法：

- 你用一个中转入口上传内容
- 中转服务再自动推送飞书群

这个中转入口可以是：

- 小程序
- 网页表单
- 自建收集机器人

优点：

- 一旦建好，体验会很顺
- 可统一格式

缺点：

- 开发成本更高
- 维护成本更高
- 第一版不建议先做

## 小程序到底能不能做

### 能做的

如果做一个小程序，它可以承担：

- 你主动把内容提交进来
- 你上传图片、视频、链接、文字
- 后端收到后再转发到飞书群或资料池

### 不能做的

小程序不能替你自动读取个人微信任意聊天记录，也不能无感监听你的微信消息。

所以：

- 小程序适合做“主动提交入口”
- 不适合做“自动读取微信聊天”的工具

## 因此，第一版不要追求“全自动”

第一版最应该追求的是：

- 你在手机上做一个很轻的动作
- 后面流程自动或半自动继续

## 最轻的中转动作

如果以“手机上最省事”为目标，建议优先级如下：

### 优先级 1：截图 / 保存 / 复制

微信里看到重要内容后：

- 文字：复制
- 图片：转存或截图
- 视频：保存到手机
- 网页：复制链接

### 优先级 2：进统一收集点

统一收集点建议暂时用：

- 本地固定目录

原因：

- 你现在已经能这么做
- 我也已经能直接处理本地文件
- 不受飞书权限阻塞

### 优先级 3：后续再同步飞书

处理完成后：

- 重要专题再同步成飞书客户版

## 当前最推荐的实际落地

### 路线 A：立即能用

`微信 -> 本地固定目录 -> 我处理 -> Obsidian/飞书输出`

这是当前最推荐路线。

### 路线 B：后续升级

`微信 -> 飞书收集群/中转服务 -> 我处理 -> 飞书/Obsidian/滴答`

这是后续升级路线。

## 为什么我更推荐先走本地目录

因为你刚才已经验证过一个事实：

- 你把截图和视频保存到本地后，我是可以直接识别和处理的

这说明这条链路已经真实可用。

相反，微信自动读取和飞书自动打通，目前都还有客观限制。

所以先把已经可用的链路变成标准动作，价值最高。

## 本地目录应该怎么设计

建议先建一个统一入口目录，例如：

- `~/Documents/Inbox`
- `~/Documents/AI输入池`
- `~/Desktop/待分析资料`

然后按最轻方式丢进去：

- 截图
- 视频
- 网页导出的 PDF
- 复制的文字说明

## 你后面的操作可以非常简单

以后你只需要：

1. 微信上看到重要东西
2. 截图 / 保存 / 复制
3. 丢进统一目录
4. 跟我说一句“处理这个目录”或“处理这个专题”

后面的事我来做：

- 提取内容
- 结构化整理
- 做专题分析
- 生成 Obsidian 主笔记
- 生成飞书客户版草稿

## 飞书在这套里的位置

飞书仍然重要，但不是微信原始输入的第一入口。

更适合的顺序是：

- 微信上来原文
- 本地目录先承接
- 分析完再进入飞书正式文档

这样更稳。

## 如果后面一定要做自动汇总

后面真正值得做的自动化是：

### 方向 1：本地目录监听

- 有新文件就提醒或整理

### 方向 2：飞书自动推送

- 分析完成后自动把结果发到飞书群或飞书文档

### 方向 3：中转上传页

- 手机端上传一个截图/链接/说明
- 后台自动入库并推送飞书

这 3 个方向都比“直接自动读微信私聊”更稳。

## 当前建议

第一版建议正式定成：

`微信是来源，本地目录是入口，飞书是汇总，Obsidian 是知识库。`

这套虽然不是最炫的，但最容易真正跑起来。
