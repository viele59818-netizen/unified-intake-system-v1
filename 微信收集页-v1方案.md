# 微信收集页 v1 方案

## 目标

做一个在微信里可直接使用的“收集口”。

你后面看到任何值得分析的内容时，只需要：

- 打开这个收集页
- 粘贴文字 / 链接
- 上传截图 / 视频
- 点提交

后面内容自动进入统一输入池，再由我继续做结构化分析。

## 为什么不把腾讯元宝作为主入口

`分享到腾讯元宝` 可以作为临时辅助，但不适合作为你的长期主入口。

原因：

- 元宝是另一个 AI 产品，不是你自己的可控收集系统
- 你很难保证后面我能稳定拿到那份原始输入
- 它更像“把内容交给别的 AI 处理”，而不是“进入你的知识系统”

所以更合理的做法是：

- 元宝可以偶尔辅助
- 但你自己的系统仍然要有一个独立收集口

## v1 的定位

第一版不要做太重，不追求聊天式交互。

只做一个足够轻的提交页，满足这几个能力：

1. 输入标题
2. 粘贴文字
3. 粘贴链接
4. 上传图片
5. 上传视频
6. 选择或填写专题名
7. 点提交

## 使用场景

### 场景 1：微信聊天内容

- 复制一段聊天内容
- 粘贴到收集页
- 补一个标题
- 提交

### 场景 2：微信截图

- 截图后直接上传到收集页

### 场景 3：客户发的视频

- 保存到手机后上传到收集页

### 场景 4：网页或产品链接

- 复制链接到收集页

## 页面结构

### 页面名称

建议名字：

- `资料收集`
- `输入收集`
- `问题收集`

我更推荐：

`问题收集`

### 页面字段

#### 1. 专题名

字段类型：

- 文本输入

示例：

- `产品舒适性问题`
- `客户A退货反馈`
- `竞品页面分析`

#### 2. 输入标题

字段类型：

- 文本输入

示例：

- `客户反馈第二版还是按不到位`
- `页面疑似误导无线使用`

#### 3. 来源类型

字段类型：

- 单选

选项建议：

- 微信
- 飞书
- 网页
- 邮件
- 图片
- 视频
- 其他

#### 4. 原始内容

字段类型：

- 多行文本

用途：

- 粘贴聊天原文
- 粘贴语音转文字
- 补一句背景说明

#### 5. 链接

字段类型：

- 文本输入

用途：

- 粘网页链接
- 粘飞书链接
- 粘产品页链接

#### 6. 图片上传

字段类型：

- 上传组件

#### 7. 视频上传

字段类型：

- 上传组件

#### 8. 备注

字段类型：

- 多行文本

用途：

- 说明这条为什么重要
- 标注是否待分析

#### 9. 提交按钮

- `提交到输入池`

## 提交后的流向

提交后建议自动落到：

```text
AI输入池/
  00_待处理/
    主题名/
      01_聊天记录/
      02_截图/
      03_视频/
      04_网页资料/
      05_补充说明/
```

## 后端建议

### 第一版建议技术路线

- 前端：微信小程序收集页
- 后端：微信云开发 / OpenCloud
- 文件存储：云存储
- 元数据：数据库

## 数据结构建议

每次提交生成一条记录：

- id
- 主题名
- 输入标题
- 来源类型
- 原始内容
- 链接
- 图片列表
- 视频列表
- 备注
- 提交时间
- 状态：待处理 / 已处理 / 已归档

## 你后面的使用动作

以后你在手机上只需要做：

1. 看到内容
2. 打开微信收集页
3. 把内容贴进去或上传
4. 提交

然后告诉我：

- “看一下这个专题”
- “整理一下今天收集的内容”
- “把这个专题做成分析”

## 我后面的处理动作

我后面负责：

1. 读取收集页提交进来的内容
2. 按专题整理
3. 输出内部主笔记
4. 输出飞书客户版
5. 拆动作到任务系统

## v1 不做的事

为了保持轻量，第一版先不做：

- 自动读取微信聊天
- 自动识别所有来源
- 自动建复杂标签体系
- 自动群聊机器人对话

## v2 可以升级的方向

第二版再考虑：

- 聊天式收集入口
- 自动专题归并
- 自动摘要
- 自动同步飞书
- 自动生成任务

## 当前最关键的设计原则

这套收集页一定要满足：

1. 在微信里能打开
2. 提交动作足够少
3. 能收文字、链接、图片、视频
4. 后面内容可被系统读取和沉淀

## 当前结论

如果你的目标是“尽量像文件助手一样方便，但先做一个能跑的版本”，那第一版最合适的形态就是：

`微信小程序收集页 + 云开发/OpenCloud 后端`

它不是最终形态，但最适合作为现在的第一步。
