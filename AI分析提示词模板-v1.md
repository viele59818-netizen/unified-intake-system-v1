# AI 分析提示词模板 v1

## 作用

这份模板用于让 AI 按固定分析框架处理输入材料，而不是自由发挥。

目标：

- 让 AI 更贴近你的思考方式
- 保持不同专题之间的分析口径一致
- 降低每次重新解释需求的成本
- 方便后续持续迭代你的个人分析框架

## 使用原则

- 先给资料，再给任务
- 先让 AI 做事实整理，再做判断
- 先让 AI 按框架输出，再让 AI 自由补充
- 重要结论必须标注“已确认”还是“推测”

## 通用主提示词

```md
请按我的固定分析框架处理这批材料，不要直接自由发挥。

你的任务分 6 步：

1. 先整理事实
2. 再做问题分类
3. 再判断可能原因
4. 再评估影响大小
5. 再判断当前是否已经闭环
6. 最后给出下一步建议

要求：

- 把“已确认事实”和“推测判断”分开写
- 不要跳步
- 不要一开始就下结论
- 如果证据不足，要明确写“证据不足”
- 如果存在多个可能原因，要区分主因和次因
- 输出尽量适合后续同步到 Obsidian 主笔记和飞书文档

输出结构固定为：

1. 事实整理
2. 问题分类
3. 原因分析
4. 影响判断
5. 是否闭环
6. 下一步建议
7. 需要补充的资料
```

## 产品问题分析版

适用于：

- 产品舒适性问题
- 售后问题
- 版本对比
- 功能体验问题
- 用户反馈汇总

```md
请按以下框架分析这批产品问题材料：

第一步，整理事实：
- 用户具体反馈了什么
- 出现在哪些场景
- 有哪些原始证据

第二步，进行问题分类：
- 功能问题
- 舒适性问题
- 质量问题
- 预期管理问题
- 使用门槛问题
- 宣传一致性问题

第三步，分析原因：
- 直接原因是什么
- 深层原因是什么
- 主因是什么
- 次因是什么

第四步，评估影响：
- 影响范围高还是低
- 是否影响退货、体验、品牌或售后压力
- 是高频还是个例

第五步，判断是否闭环：
- 已解决
- 部分解决
- 尚未解决
- 证据不足

第六步，输出建议：
- 现在最应该验证什么
- 现在最应该改什么
- 是否需要同步客户版文档

补充要求：
- 结构问题和功能问题分开看
- 用户感知问题和真实功能问题分开看
- 页面理解偏差和产品本身问题分开看
```

## 售后反馈汇总版

```md
请帮我处理一批售后反馈，要求不要直接写泛泛总结，而是按下面步骤输出：

1. 提取高频原话模式
2. 合并同类问题
3. 标出最核心的 3 个问题
4. 说明这些问题更像是结构问题、功能问题还是预期管理问题
5. 判断哪些问题最可能导致退货
6. 输出一版适合内部讨论的结论

要求：
- 尽量保留用户真实表达的意思
- 不要过度润色到失真
- 要区分高频问题和偶发问题
```

## 飞书客户版整理版

```md
请把下面的内部分析内容，整理成适合给客户或协作方看的正式版本。

要求：
- 语气客观克制
- 保留结论，但不要保留内部过于主观的推测
- 结构清晰，方便直接放入飞书文档
- 优先写“问题背景、核心判断、当前状态、风险、下一步建议”
- 如果证据不足，不要写成已经确认

输出结构：
1. 问题背景
2. 当前反馈汇总
3. 核心问题判断
4. 当前版本评估
5. 风险与说明
6. 下一步建议
```

## Obsidian 主笔记整理版

```md
请把这批输入整理成适合进入 Obsidian 主笔记的内容。

要求：
- 优先保留原始证据结构
- 保留未确认点
- 保留待验证项
- 方便后续再抽取成飞书客户版

输出结构：
1. 原始输入摘要
2. 事实整理
3. 问题分类
4. 原因分析
5. 当前结论
6. 下一步动作
7. 待补资料
```

## 输入层清洗版

```md
请先不要做深度结论，只对这批输入材料做清洗和整理。

要求：
- 每条输入提炼一句摘要
- 标出来源类型
- 标出时间
- 标出初步标签
- 判断证据价值高、中、低
- 标出是否建议进入主笔记

不要做大段分析，不要提前给产品结论。
```

## 使用建议

### 第一轮

用“输入层清洗版”或“Obsidian 主笔记整理版”。

### 第二轮

用“产品问题分析版”。

### 第三轮

用“飞书客户版整理版”。

## 后续迭代方向
