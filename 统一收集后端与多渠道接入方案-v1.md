# 统一收集后端与多渠道接入方案 v1

## 目标

搭一套统一的“输入收集后端”，让不同渠道的原始内容最终都汇到同一个地方，再由 AI 和后续工作流继续处理。

这套方案要解决 3 件事：

1. 哪个入口先做最容易跑起来
2. 后面如何兼容飞书、企业微信、网页等渠道
3. 架构上如何避免后面重做

## 核心结论

第一版最合适的路线不是从个人微信硬接，而是：

`统一收集后端 + 飞书/企业微信先接入 + 个人微信后续再接`

原因：

- 飞书和企业微信更容易做正式接入
- 官方能力更清晰，机器人、Webhook、自建应用路径更成熟
- 后续扩到网页、表单、收集页也更容易
- 不会一开始卡死在个人微信接入这件事上

## 一句话架构

`飞书 / 企业微信 / 网页收集页 / 本地导入 -> 统一收集后端 -> 输入池 -> AI分析 -> Obsidian / 飞书文档 / 滴答清单`

## 第一版最推荐的接入顺序

### 第一阶段：先搭统一后端

先不纠结所有入口，而是先把后端收集能力建好。

后端先能支持：

- 接收文本
- 接收链接
- 接收图片
- 接收视频
- 接收来源标记
- 接收专题名

### 第二阶段：先接飞书和企业微信

先接这两个渠道的原因：

- 都更适合正式工作流
- 更容易跑通自动化
- 更适合后续对接文档与协作

### 第三阶段：再评估个人微信接入

个人微信应放到第二阶段之后再考虑。

更适合作为：

- 辅助入口
- 中转入口
- 聊天式体验层

而不是第一版核心依赖

## 为什么个人微信不适合作为第一版核心

### 技术原因

- 官方开放能力不适合直接做个人聊天自动读取
- 很多“个人微信机器人”方案依赖非官方实现
- 稳定性、合规性和维护成本都更差

### 产品原因

- 你真正需要的是“能持续跑的系统”
- 不是“勉强做出一个能演示的入口”

### 架构原因

- 如果一开始把系统强绑定在个人微信方案上，后面很容易整体受限
- 先把后端打稳，后面不管接什么入口都能共用

## 第一版系统结构

### 1. 输入接收层

负责接入不同来源的原始内容。

第一版建议支持：

- 飞书
- 企业微信
- 网页收集页
- 本地导入

后续再扩：

- 个人微信中转

### 2. 统一收集后端

负责把所有入口内容转成统一结构。

建议统一结构字段：

- id
- topic
- title
- source_channel
- source_type
- raw_text
- links
- image_assets
- video_assets
- note
- created_at
- status

### 3. 输入池

相当于待处理原始资料仓。

状态建议：

- 待处理
- 处理中
- 已归档

### 4. 分析与输出层

后续由我继续处理：

- AI 结构化整理
- Obsidian 主笔记
- 飞书客户版
- 滴答执行任务

## 推荐的第一版技术选型

## 方案 A：轻量 Web 服务

适合先验证流程。

建议：

- 前端：简单表单页 / 收集页
- 后端：Node.js
- 存储：JSON 文件或 SQLite
- 文件：本地目录 / 对象存储

优点：

- 开发快
- 调试方便
- 最适合做第一版

缺点：

- 后续接渠道还需要继续补

## 方案 B：云开发型后端

适合后续转向微信生态。

建议：

- 前端：微信小程序
- 后端：微信云开发 / OpenCloud
- 数据：云数据库
- 文件：云存储

优点：

- 更适合手机端收集
- 更适合后续小程序演进

缺点：

- 起步门槛略高
- 第一版开发和调试成本更高

## 当前建议

如果目标是“最快跑起来”，建议：

### v1

- 先做轻量 Web 后端
- 把统一数据结构和输入池跑通

### v2

- 再接微信小程序 / OpenCloud

这样做更稳。

## 多渠道接入如何设计

关键不是每个渠道怎么单独做，而是：

每个渠道最后都转成统一 payload。

建议统一 payload：

```json
{
  "topic": "产品舒适性问题",
  "title": "客户反馈第二版还是按不到位",
  "source_channel": "wechat",
  "source_type": "chat",
  "raw_text": "客户表示第二版佩戴后按摩点还是碰不到有效区域",
  "links": [],
  "note": "可能属于贴合问题",
  "assets": []
}
```

只要后端吃这套格式，后面：

- 飞书
- 企微
- 网页
- 小程序
- 本地导入

都只是不同入口，不是不同系统。

## 第一版推荐支持的入口

### 飞书入口

建议能力：

- 群消息中转
- 文档提交
- 后续接 `lark-cli`

### 企业微信入口

建议能力：

- 群机器人 Webhook
- 自建应用消息接收

### 网页 / 表单入口

建议能力：

- 手动粘贴文字
- 上传图片/视频
- 填写专题和标题

### 本地导入入口

建议能力：

- 从目录扫描资料
- 批量导入截图、视频、文本

## 为什么企业微信比个人微信更值得先做

- 官方路径更明确
- 机器人/Webhook 更成熟
- 更接近“聊天式接收”
- 更适合正式自动化

所以如果你以后真的想要“像聊天一样发过去”，企业微信会比个人微信更适合作为第一版聊天入口。

## 第一版开发优先级

1. 定义统一数据结构
2. 建统一收集后端
3. 建输入池
4. 接一个网页收集入口
5. 接飞书或企业微信入口
6. 再做后续 AI 分析联动

## 和你当前系统的关系

后端接起来之后，后面流向不变：

- 输入进后端
- 我做分析
- 输出到 Obsidian
- 输出到飞书客户版
- 拆到滴答清单

所以它不是另起一套，而是补齐你现在最前面的“收集层”。

## 实际建议

如果现在就要定方向，我建议这样定：

### 第一版产品目标

不是“搞定个人微信自动化”，而是：

`先做一个可扩展的统一收集后端`

### 第一版入口目标

不是“所有渠道都通”，而是：

- 先通网页/表单入口
- 再通飞书/企业微信其中一个正式入口

### 第二版目标

再评估微信聊天式入口

## 当前结论

最容易跑起来、又不容易后面推翻的路线是：

`统一收集后端先行，飞书/企业微信优先接入，个人微信后续再接。`

这条路最稳，也最适合你后续扩到多个渠道。
