# 部署方案 v1

## 目标

把当前的收集原型变成一个可通过 `https://` 访问的正式入口，这样后面才能在微信里直接打开使用。

## 当前状态

当前已经具备：

- 一体化前后端原型
- 本地可运行
- 前端由后端直接托管
- API 和页面同域

这意味着后面上线只需要部署一个服务，不需要拆前后端。

## 最小上线条件

要在微信里真正可用，至少需要：

1. 一个公网地址
2. `https://`
3. 服务能持续运行

## 第一版最推荐的平台

如果目的是“尽快上线测试”，我建议优先考虑：

### 方案 A：Render

优点：

- 最容易上手
- Node 服务部署简单
- 支持 `render.yaml`
- 适合先做测试环境

### 方案 B：Railway

优点：

- 部署也很轻
- 对 Node 友好

### 方案 C：你自己的服务器

优点：

- 自主可控

缺点：

- 配置更重

## 当前代码已经为部署做的准备

- 默认监听 `0.0.0.0`
- 提供 `package.json`
- 提供 `render.yaml`
- 提供健康检查接口 `/health`

## 最推荐的第一版动作

### 1. 把项目放到 Git 仓库

最少需要包含：

- `intake-backend-v1/`
- `weixin-intake-v1/`

### 2. 在部署平台创建一个 Node Web Service

启动命令：

```bash
npm start
```

根目录：

```text
intake-backend-v1
```

### 3. 部署后拿到一个 https 地址

例如：

```text
https://your-intake-service.onrender.com
```

### 4. 用手机微信打开测试

重点测试：

- 页面能否打开
- 是否能提交文字
- 是否能上传图片
- 是否能上传视频

## 上线后的用途

部署成功后，这个地址就会变成：

- 一个手机可打开的收集口
- 一个微信里可直接点开的入口
- 后面可继续升级成小程序前的验证版本

## 当前建议

第一版不要急着上复杂云开发。

先把这个网页版本部署成一个可访问的 `https://` 地址，先确认：

- 你是否真的愿意用
- 在微信里打开是否顺手
- 提交流程是否足够轻

如果这一步顺了，再决定要不要进入小程序版。
